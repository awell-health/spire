# v0.56.0

## Fixes

- **A wizard no longer spawns duplicate apprentices after the host
  sleeps** — when a laptop running `spire up` sleeps mid-epic, the
  wizard and apprentice `claude` sessions die. The daemon orphan-sweep
  (dead PID plus a heartbeat that went stale against wall-clock while
  suspended) reopens the epic, and a fresh wizard re-dispatches. Because
  `ComputeWaves` re-includes still-in-progress children and the dispatch
  path had no idempotency guard, the new wizard spawned a second
  apprentice alongside any survivor — wasting work and, in some cases,
  committing against the wrong branch. The wave, sequential, and direct
  dispatch paths now snapshot live agents (once per wave) before
  spawning and refuse to start a second apprentice for a bead that
  already has a live one, parking the step rather than silently
  advancing the epic over an in-flight child. The guard keys strictly on
  a live process, so a genuinely dead post-sleep entry still allows the
  needed re-spawn, excludes the wizard's own registry entry, and fails
  open on a registry-read error so it never blocks legitimate dispatch.

- **Epic subtasks no longer risk committing to `main` instead of the
  epic's base branch** — a child of an epic that overrides its base
  branch (`spire file --branch …`) could reach the implementation path
  without its `base-branch:` label and fall back to committing against
  `main`. Label propagation during planning now fails loud (with one
  retry) instead of warning, and the resume/enrich path backfills the
  label for any override-epic child that is missing it. Default-branch
  epics are entirely unaffected.

## Notes

- This release addresses the duplicate-process and wrong-branch symptoms
  of host sleep. The underlying wall-clock heartbeat staleness that lets
  a long sleep trigger the orphan-sweep in the first place is unchanged
  — the dispatch guard neutralizes the harmful outcome regardless.
