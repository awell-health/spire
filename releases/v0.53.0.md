# v0.53.0

## Fixes

- **The desktop Logs tab now opens on the most recent attempt** — the bead
  logs list (`GET /api/v1/beads/{id}/logs`) returned artifacts ordered by
  `attempt_id`, which is a random short code (e.g. `spd-th36`), not a
  chronological value. The tab defaults to `artifacts[0]`, so it always
  opened on an arbitrary *old* attempt and never the latest run. The store
  query (`ListLogArtifactsForBead`) now orders attempt groups
  most-recent-first by their earliest artifact timestamp via a window
  function (`MIN(created_at) OVER (PARTITION BY attempt_id) DESC`), while
  keeping each attempt's rows contiguous and in ascending sequence order so a
  single attempt still reads top-to-bottom. Ordering stays fully
  deterministic (`id ASC` final tiebreak), so the gateway's offset-based logs
  cursor remains stable across pages. Verified against a live Dolt engine, not
  just sqlmock.

## Improvements

- **Terminal escalations now carry the actual failure on the bead** — when a
  formula routes to a terminal `bead.finish status=escalate` step (e.g. an
  epic's `implement-failed`), the step has no handle on the upstream failure
  detail: it fires on a *declared-but-bad outcome* (`outcome != verified`),
  not a hard error, so the only thing the escalation comment could say was the
  content-free `bead-finish-escalate — formula requested escalation`. The real
  reason — for instance a build/test failure surfaced by `verify.run`, like an
  integration test that can't reach a dependency (`ECONNREFUSED`) — lived only
  in `~/.local/share/spire/wizards/wizard-<bead>.log`, leaving the archmage to
  diagnose a stuck epic by hand. The escalate branch now appends the tail
  (last 40 lines) of the run's wizard log to the bead as an
  `Escalation diagnostics` comment, so the failure that actually blocked the
  bead is visible where you read it. The lookup reuses the agent backend's log
  naming and is fully best-effort — it never perturbs the escalation path.

  This is an interim stopgap. The durable fix is to propagate the failing
  upstream step's failure summary as a first-class output that the terminal
  escalate step reads, rather than scraping the run log.
