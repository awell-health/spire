# v0.54.0

## Fixes

All four changes in this release target one diagnosis: the shared dolt
sql-server sitting at ~150% CPU on an otherwise idle tower, serving
~6,400 queries/sec. Attribution (measured by pausing each client while
sampling the server's cputime and `Questions` counter): ~120 points
came from the desktop gateway's board polling, ~30 from the steward's
cycle loop.

- **Board listings no longer issue one wisp lookup per bead** — the
  beads library's bulk dependency fetch partitions issue IDs into
  wisp vs permanent tables with `SELECT 1 FROM wisps WHERE id = ?`
  once per ID. With 15k+ beads, every `GET /api/v1/beads` poll fired
  15k+ point queries before fetching a single dependency row — the
  single hottest statement on the server (in flight in 17 of 25
  processlist samples). `store.PopulateDependencies` now queries both
  dependency tables directly with batched 200-ID `IN` clauses when the
  store exposes its raw `*sql.DB`; for a bulk read the per-ID routing
  is unnecessary since an ID's rows live in exactly one table. The
  beads-library path remains as the fallback for stores without raw
  DB access.

- **The board listing is bounded** — `GET /api/v1/beads` returned every
  bead in the tower: all statuses including closed (98% of rows on a
  mature tower) plus internal engine types (`message`, `step`,
  `attempt`, `review` — more than half of all rows). One poll scanned
  and serialized 15,768 wide rows into 9.6MB of JSON taking ~8s.
  Internal types are now excluded at the SQL level, and closed beads
  are fetched only from the last 14 days (`?closed_days=N` overrides
  the window, `0` disables closed beads, and an explicit `?status=`
  keeps full-history semantics).

- **Unchanged board polls are served from cache** — the gateway now
  memoizes `/api/v1/beads` responses keyed by query string and
  `dolt_hashof_db()`, a single cheap query that hashes the working
  set. While the database hash is unchanged, polls cost one query
  instead of a full re-list, and clients sending `If-None-Match` get
  a `304` with no body at all (responses now carry an `ETag`).

- **The steward guarantees idle time between cycles** — the cycle loop
  used a ticker, which keeps one tick buffered while a cycle runs. The
  default `--interval 10s` with ~26s cycles meant the next cycle
  started the instant the previous one finished, indefinitely — the
  steward never idled. The loop now re-arms a timer only after a cycle
  completes, making the interval a real pause rather than a cadence
  the loop can saturate.

- **`ResolveRepo` no longer forks a `dolt sql` subprocess per call** —
  every shared-repo lookup forked a full dolt CLI (which itself forks
  a `dolt send-metrics` child) and opened a fresh server connection,
  ~17 new connections/sec under steward load, to re-read a
  `repos`-table row that never changes mid-run. The lookup now goes
  through the warm in-process store pool when one is open
  (parameterized query, no subprocess) and rows are memoized per
  (database, prefix) with a 5-minute TTL; the subprocess path remains
  as fallback for processes without an open store.

## Notes

- The gateway fixes take effect in whichever binary serves the desktop
  API (`spire serve`). The desktop app bundles its own spire — until
  the bundle is updated to this version, the board-polling load from
  the old bundled binary persists.
