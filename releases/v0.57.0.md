# v0.57.0

## Performance

This release eliminates the two distinct causes of the shared Dolt
sql-server pinning the CPU on an otherwise quiet tower — an always-on
**read** storm and a per-operation **write** storm. Both were diagnosed
by pausing each client (SIGSTOP) while sampling the server's cputime and
query counters, then pinned with per-step instrumentation.

- **The steward no longer scans the whole tower every cycle (read storm).**
  `lifecycle.DispatchableBeads` listed every non-closed / non-filed bead —
  on a mature tower that is hundreds of long-resident internal `step`
  beads — and then issued a per-candidate child lookup that the beads
  library fanned out into ~22 `SELECT`s each: roughly 2,000 queries per
  ~13s cycle, over 96% of all Dolt selects, holding the server around
  150–230% on an idle tower. The candidate query now excludes internal
  bead types (`message`/`step`/`attempt`/`review`) — the steward already
  dropped them immediately afterward via `store.IsWorkBead`, so the
  dispatch result is unchanged — and step-bead readiness is now fetched
  for all candidates in a single batched join (`store.GetStepBeadsBatch`)
  instead of one lookup per candidate. Per-cycle child-query count drops
  from ~99 to a handful. (Pausing the steward took Dolt from ~4,000
  selects / 30s to 2.)

- **`spire` no longer re-imports the bead database on every `bd`
  shell-out (write storm).** Each `bd` CLI invocation against a populated
  tower re-ran the beads library's startup auto-import of the full ~24MB
  `.beads/issues.jsonl` into an "empty database" — a ~36,000-row no-op
  upsert (plus a full JSONL rewrite) that then failed validation on
  Spire's custom `ready` status. Spire shelled out to `bd` on several
  hot or frequent paths, each paying that cost in full:

  - `spire up` custom type / status setup (~73k inserts per `up`),
  - every agent run — the metrics recorder's `agent_runs` insert
    (~36k per run, multiplied by concurrent wizards/apprentices into the
    largest CPU spikes),
  - the wizard's per-run bead-title lookup,
  - the `spire cost` / `metrics` / `trace` / `attempt` / `grok` /
    `focus` / `logs` / `lifecycle status` report queries, and
  - `spire pull` / `push` / `sync` remote configuration.

  All of these now go through the in-process store (or the `dolt` binary
  for remote config), which talks to the live, already-populated server
  directly — no subprocess, no re-import. Measured on a live tower:
  `spire up` dropped from **73,112 → 17** inserts, and a single
  `spire execute` (agent run) from **36,717 → 165**.

## Notes

- The two fixes are independent. Together they take an idle tower's Dolt
  CPU from a sustained ~150–230% down to roughly 0% between cycles, and
  remove the multi-hundred-percent spikes seen during `spire up` and
  wizard/apprentice runs.
- `bd init` (used by `spire tower create` and cluster bootstrap) still
  imports the JSONL — that is the correct one-time hydration of a brand
  new database, not a no-op. Manual `bd` CLI commands (e.g.
  `bd update`) are unchanged and still talk to the server via the CLI.
