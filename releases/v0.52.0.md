# v0.52.0

## Fixes

- **Steward/daemon no longer churn Dolt connections every cycle** — `TowerCycle`
  and `DaemonTowerCycle` used to call `store.OpenAt()` + `defer store.Reset()`
  on every cycle, which tore down and rebuilt the Dolt connection pool each time.
  Because cycles run effectively back-to-back, this defeated pooling entirely:
  every cycle started with a cold pool, and each new connection pays a
  server-side session-bootstrap cost, so the churn pinned several CPU cores on
  the `dolt sql-server` even when little real work was happening. Both paths now
  use a new warm per-tower store cache (`store.UseTowerStore`) that opens each
  tower's store once and reuses the pool across cycles. The cache health-checks
  the connection on reuse and transparently reopens if the Dolt server was
  restarted, preserving the reconnect-on-restart behavior the old per-cycle
  `Reset()` provided incidentally.

## Improvements

- **Connection-source logging behind `SPIRE_DEBUG_CONN`** — set the env var to a
  non-empty value to log every raw Dolt connection open (with a stable site
  label and a running counter) plus warm-cache hit/miss/stale events. Lets you
  attribute a connection storm to its exact call site and confirm the rate has
  dropped, without re-deriving the diagnosis. Instrumented sites: the warm
  tower-store cache, `steward.getDBForRouting`, `daemon.syncToOLAP`, and
  `executor.NewDoltGraphStateStore`.
- **Warm tower-store cache is cleaned up on shutdown** — `spire steward` and
  `spire daemon` close all cached per-tower pools (`store.CloseTowerStores`) on
  `SIGINT`/`SIGTERM` (and after `--once`).
