Good — `StepConfig = formula.StepConfig` is a type alias in deps.go. The flow is: `EscalateGraphStepFailure` receives `stepCfg.Flow` at the call site in graph_interpreter.go:217. This `flow` value is included in the `nodeCtx` string but not extracted as a separate metadata field.

Now I have all the context for a precise plan. Here it is:

---

**Approach:** The recovery agent stores graph step names (e.g., `verify-build`) as `SourceStep` in recovery metadata, which flows into `RetryRequest.FromStep`. The wizard rejects these because it only knows its own phase names (`design`, `implement`, `commit`, `build-gate`, `test`, `review`). Fix this by: (1) also storing the graph step's `flow` field in recovery metadata, (2) using it as the wizard-compatible phase in `handleVerify`, and (3) adding a fallback mapping in the wizard for robustness.

**Key files:**
- `pkg/executor/recovery_protocol.go` — add `MapToWizardPhase()` function and `KnownWizardPhases` set
- `pkg/executor/recovery_phase.go` — call `MapToWizardPhase()` on `failedStep` before setting `FromStep` in `handleVerify`
- `pkg/wizard/wizard_retry.go` — use `executor.MapToWizardPhase()` instead of raw `knownWizardSteps` lookup in `checkRetryRequest`
- `pkg/recovery/keys.go` (or wherever `KeySourceStep` is defined) — add `KeySourceFlow` constant for the new metadata field
- `pkg/executor/executor_escalate.go` — extract `flow` from `nodeCtx` in `buildSeedMetadata()` and store as `SourceFlow`; also seed `SourceFlow` in `RecoveryMetadata`
- `pkg/executor/recovery_context.go` — read `SourceFlow` into `FullRecoveryContext` and use it to populate a new `WizardPhase` field
- `pkg/executor/recovery_protocol_test.go` — add tests for `MapToWizardPhase`
- `pkg/wizard/wizard_retry_test.go` — update tests for the new mapping behavior

**Steps:**

1. **Add `KeySourceFlow` to recovery metadata keys** (pkg/recovery/keys.go or equivalent). Add a `source_flow` metadata key alongside the existing `source_step` key.

2. **Extend `RecoveryMetadata` and `buildSeedMetadata`** (pkg/executor/executor_escalate.go). In `buildSeedMetadata()`, also extract `flow=<value>` from `nodeCtx` (same pattern as `step=`). Store it in a new `SourceFlow` field on `RecoveryMetadata`. This captures the formula step's `flow` attribute (e.g., `implement`, `task-plan`, `epic-plan`) at escalation time.

3. **Add `WizardPhase` to `FullRecoveryContext`** (pkg/executor/recovery_context.go). In `BuildRecoveryContext()`, after reading `FailedStep` from `KeySourceStep`, also read `SourceFlow` from metadata into a new `WizardPhase string` field on `FullRecoveryContext`.

4. **Add `MapToWizardPhase()` in `recovery_protocol.go`**. This is the translation function. Logic:
   - If step is already in `KnownWizardPhases` → return as-is
   - Check a static mapping for known graph step names:
     - `verify-build`, `build-failed` → `build-gate`
     - `dispatch-children` → `implement`
     - `design-check` → `design`
     - `implement-failed` → `implement`
     - `merge`, `close`, `discard` → `review` (retry from review to redo the terminal flow)
     - `plan`, `materialize` → `design`
     - `sage-review`, `review-fix`, `arbiter` → `review` (subgraph-review steps)
   - Final fallback → `implement` (safest restart point for the wizard)
   - Export `KnownWizardPhases` so the wizard package can reference it.

5. **Use `MapToWizardPhase` in `handleVerify`** (recovery_phase.go ~line 1430). After determining `failedStep`, also check `fullCtx.WizardPhase` (the `flow` from step metadata). Prefer `WizardPhase` if non-empty, else apply `MapToWizardPhase(failedStep)`. Set the mapped value as `FromStep`.

6. **Use `MapToWizardPhase` in wizard's `checkRetryRequest`** (wizard_retry.go). Replace the direct `knownWizardSteps[req.FromStep]` check with `mapped := executor.MapToWizardPhase(req.FromStep)`. If `mapped` differs from `req.FromStep`, log the translation. Update `req.FromStep` to the mapped value before proceeding. Remove the local `knownWizardSteps` var (it's now canonical in executor).

7. **Tests:**
   - `recovery_protocol_test.go`: Test `MapToWizardPhase` — known wizard steps pass through, graph step names map correctly, unknown names fallback to `implement`.
   - `wizard_retry_test.go`: Add a test case where `FromStep="verify-build"` — should translate to `build-gate` and succeed instead of rejecting.
   - `executor_escalate_test.go` (if exists): Test that `buildSeedMetadata` extracts `flow` into `SourceFlow`.
   - `recovery_context_test.go`: Test that `WizardPhase` is populated from metadata.

**Edge cases / gotchas:**
- **Graph steps with no `flow` attribute** (like `verify-build`, `dispatch-children`): The `SourceFlow` metadata will be empty. The static mapping in `MapToWizardPhase` handles these by name.
- **Subgraph steps** (from `subgraph-review`, `subgraph-implement`): These have step names like `sage-review`, `review-fix`, `verified`. The static map must cover these, and the fallback must be sensible.
- **Empty `failedStep`**: If both `FailedStep` and `WizardPhase` are empty, `handleVerify` should still set a `FromStep`. Use `implement` as default — the wizard will restart from the implement phase.
- **Circular dependency risk**: `MapToWizardPhase` must live in `pkg/executor` (not `pkg/wizard`) since wizard already imports executor. The wizard calls `executor.MapToWizardPhase()`.
- **Recovery protocol test** at line 16 uses `FromStep: "verify-build"` — this is the exact bug scenario. Update it to verify the mapping works end-to-end.
- **The `flow` values in formulas** (`task-plan`, `epic-plan`, `sage-review`, `review-fix`, `arbiter`) don't match wizard phases directly. The `MapToWizardPhase` function should also map these: `task-plan`/`epic-plan` → `design`, `sage-review` → `review`, `review-fix` → `review`, `arbiter` → `review`, `implement` → `implement`.

- `go test ./pkg/wizard/... -run TestRetry` — verify the wizard accepts translated step names
- `go test ./pkg/executor/... -run TestRecovery` — verify end-to-end recovery context building
